# 主題五：勞動福祉 — 3 提案

> 競賽：2026 雙北程式設計節城市儀表板大黑客松
> 評分：受眾洞察 40 / 雙北獨特性 30 / Demo & 故事性 30
> 核心願景：看 → 決策 → 自動輔助
> 硬性條件：≥4 雙北組件（含 ≥1 地圖）、組件下拉切換 台北/雙北、Apexcharts + Mapbox、AI 模型 llama3.3-ffm-70b-16k-chat（30 RPM）

**差異化策略**：
- #1 = OD 流動故事敘事（社會議題 framing）
- #2 = 市民端 AI 助理（NL 補助資格試算，行動入口）
- #3 = 中高齡 + 身障再就業（弱勢 persona，行動匹配）

三個提案分別覆蓋「**勞動條件不平等**」「**福利資源資訊落差**」「**就業弱勢匹配缺口**」，避免全部聚焦於老人照護或單純失業率報表，刻意與既有 dashboard（育兒支持/長照據點/扶養比/高齡就業/全市年齡分區）做明確區隔。

## 提案 #1: 雙北跨城血汗線 — 通勤勞動條件落差地圖

**Pitch (一句話)**：
住新北、上班北市的「跨城通勤族」每天多耗 1.5 小時通勤，卻常落入勞動條件較差的行業；本儀表板把雙北就業 OD、違法雇主、勞檢、職場申訴疊在同一張地圖上，讓政府看見「跨城外溢」造成的隱性過勞，讓市民知道自己是不是踩進地雷雇主。

**核心受眾**：
- 主：住新北通台北上班的中下階層勞工（板橋/三重/中和/新莊/汐止）
- 次：勞動局督察/政策局處（決策面）、媒體（衝擊面）

**雙北痛點**（不可替換的雙北議題）：
1. 新北勞動人口大量外流台北 → 通勤族集中在低薪服務業/物流/餐飲
2. 違法雇主資料雙北各自公告，跨城求職者沒有單一視角
3. 北市勞檢密度 ≠ 新北勞檢密度，跨城雇主鑽漏洞
4. 申訴與性別工作平等法案件分散兩市資料源，看不見系統性風險

**核心價值**：
把「個人血汗故事」放回「雙北空間結構」中，讓抽象的勞基法違法統計變成「我該不該換工作」的具體判斷。

**Demo 衝擊力（2 分鐘 pitch）**：
1. **Hook（0:00–0:20）**: 大螢幕打出 persona「阿芳，三重→信義區百貨樓管，月薪 32k，每天通勤 1h40m」。地圖出現一條從三重亮到信義的紅色 OD 流動線。
2. **組件切換（0:20–1:00）**: 下拉切換到「雙北」模式，畫面同步浮現 12 條最大跨城 OD 流動線；右側 Apexcharts 堆疊長條顯示「跨城職業 vs 留在原市職業」的行業差異 → 跨城族集中在批發零售/住宿餐飲/支援服務（低薪三業）。
3. **AI / Insight（1:00–1:30）**: 點擊任一 OD 流線，AI 助理朗讀 「住新北通台北的批發零售業勞工，2024 雙北合計勞檢違法率為 X%，較留在新北本地工作高 Y%」並顯示熱區 top3 違法雇主聚集點（去識別化匿名化呈現「行業聚集地」）。
4. **行動入口（1:30–2:00）**: 切換到市民模式，輸入「我在板橋住、信義區做服務業，老闆要我每天加班 3 小時不給加班費」→ AI 比對勞檢公告 + 勞基法 84-1 條，回覆「您的雇主是否在違法名單／可申訴的最近窗口」並導航最近就業服務站 / 勞動檢查處。

**雙北對比角度**：

- 新北性別工作平等違法案 VS 北市職場性騷擾申訴件數的成長率對比

**AI 應用（llama3.3-ffm-70b-16k-chat，30 RPM 內）**：
1. **「我的雇主是不是地雷？」自然語言查詢**：使用者輸入公司名 + 行業 + 行為描述 → 模型 (a) 對 違法雇主 CSV 做檢索（embedding 預先建好），(b) 比對勞基法條文（RAG），(c) 回覆風險等級 + 申訴管道。
2. **OD 故事敘事 narrator**：點擊地圖任一流線時，AI 即時用 1 段 80-120 字描述「這條通勤線背後的勞動結構」（避免乾燥報表感）。
3. **Rate limit 設計**：使用者前端 debounce 2s + 後端 token bucket 30 RPM，超出時 fallback 顯示 cached canned reply。

**可解釋性**：
- 違法雇主資料只顯示「行業群聚熱區」+「公告日期」，不顯示個別公司名稱於地圖（避免誤傷小店家），但點擊熱區可彈出官方公告原始 PDF 連結。
- 風險等級用顏色 5 階 + 文字說明每個等級對應的勞基法條文。

**行動入口**：
- 「立即申訴」按鈕：依使用者地址自動匯出最近的勞動局申訴窗口
- 「找職缺」按鈕：跳轉北市/新北就業服務處職缺資訊（北市 `67e8ddde-3cbb-4946-bf16-73774abcd1b9` + 新北「政府所屬就業服務據點」`4427DB9F-2EB0-4646-A291-E6031D564C4F`）
- 「分享我的故事」匿名表單，未來可作群眾外包資料

**與既有 dashboard 差異**：
- 既有有「育嬰假性別比」「高齡就業人口年增結構」「全市年齡分區」「扶養比」，但**沒有任何 OD 通勤×勞動條件交叉組件**。
- 既有違法雇主只是清單，本提案首次將其「空間化 + 時序化 + 行業化」。

**資料品質風險**：
- OD 流動需從通勤普查／健保投保資料推估，data.taipei 沒有直接 OD 資料 → mitigation：用「就業者行業結構」+「行政區性別人口」推估（雙北主計處資料）。Demo 階段以 2024 主計處推估值疊加，並標示「估算」浮水印。
- 違法雇主名單為「不更新」或「每日」更新差異大 → 顯示資料快照日期。

**技術可行性**：
- 所有資料 CSV/JSON 直接抓取，雙北各 6-8 個資料集；地圖層用 Mapbox GL JS 弧線 (arc-layer) + heatmap-layer。
- 風險：OD 弧線渲染效能（>500 條時掉幀）→ 限制顯示 top 30 流動線。

**亮點 scoring（1-5）**：
- 受眾洞察：5（雙北通勤族故事鮮明）
- 雙北獨特性：5（OD 跨城是雙北黑客松最該被回答的議題）

## 提案 #2: 「補助雷達」— 雙北福利資格自然語言試算與最近資源導引

**Pitch (一句話)**：
台北 / 新北的勞動與社福補助條件不同步，市民根本搞不清楚自己「符合什麼補助 / 該去哪裡辦」；本儀表板讓市民用自然語言講一段自己的處境，AI 即時試算雙北所有可申請補助 + 在地圖上導引最近受理據點。

**核心受眾**：
- 主：求職青年、單親媽媽、低收入戶、中高齡轉職者、育兒家長、外籍移工
- 次：里幹事 / 社工（前線轉介工具）

**雙北痛點**：
1. **同樣身分、不同補助**：北市生育獎勵、托育補助和新北的條件、金額、年齡上限都不同步，跨市搬家就要重新查
2. **補助文件碎片化**：北市福利散在勞動局、社會局、衛生局；新北「政府福利補助資訊」每日更新（`73C24D1C-E1F0-44CE-8A31-CF835EF37E74`）但欄位純文字非結構化
3. **里幹事憑經驗轉介**：沒有單一視窗能比對「此人 vs 雙北所有補助」
4. **跨城求助繞遠路**：住新北但工作地在北市，到底該回新北戶籍地辦理還是北市辦理，混亂

**核心價值**：
把雙北 200+ 種補助制度「個人化試算 + 視覺化最近據點」，從報表變成行動工具。

**Demo 衝擊力（2 分鐘 pitch）**：
1. **Hook（0:00–0:20）**: persona「阿玲，板橋單親媽媽，1 個 3 歲小孩、1 個小一，去年離職現在打零工、月入 22k」，輸入到聊天框。
2. **AI 試算（0:20–1:00）**: AI 即時跑完雙北 50+ 種主要補助，回傳一張卡片清單：「您可能符合：① 新北中低收入戶生活扶助 ② 新北托育補助 ③ 北市就業服務處職訓津貼（您工作地在北市可申請）④ 新北單親家庭補助；不符合：⑤ XX 因條件年齡」。每張卡片一行原因 + 一個「立即查辦理流程」按鈕。
3. **地圖導引（1:00–1:30）**: 右側雙北地圖同時亮出阿玲家附近 5 個受理據點（社會局福利服務中心、就業服務站、區公所、托育資源中心、兒童托育中心），按距離排序、顯示開放時間。
4. **進階（1:30–2:00）**: 切到「政策局處模式」：地圖換成「雙北補助申請熱區」分布；下拉雙北切換看「同一身分（單親）在雙北能領的補助總額」對比 → 衝擊力呈現政策落差。

**雙北對比角度**：
- 同 persona「板橋單親媽 vs 中和單親媽 vs 大同區單親媽」可領補助總額對比
- 新北 vs 北市同類型補助資格門檻差異（用 AI 自動萃取，視覺化矩陣）
- 補助資金來源（公務預算 / 公益彩券盈餘 / 身心障礙者就業基金）的雙北結構差異

**AI 應用**（核心亮點）：
1. **NL → 結構化資格判斷**：使用者描述（自由文）→ llama3.3 萃取「年齡/家戶人數/收入/身分別/居住地/就業狀態」7-8 個結構化欄位 → 與 RAG 出的補助條文做規則比對 → 輸出符合度 0/1/部分符合 + 缺什麼條件。
2. **RAG 知識庫**：先把雙北補助條文 chunk 化（每條補助一個 chunk），用 embedding 預建索引；NL query 先檢索 top-5 補助再喂給 llama3.3 做試算。
3. **Rate limit 設計**：30 RPM → 每位使用者前端 cool-down 5s + cache 相同 query 1 小時，避免 demo 時並發爆 API。
4. **可解釋**：每個試算結果旁邊一定附「依據條文：XX 第 X 條」連結到原始 dataset 條目。

**可解釋性**：
- 每個補助卡片顯示「資料來源 dataset 名稱 + 公告/更新日期」

- 備案：AI 不確定時 fallback 到人工窗口聯絡資訊，不裝懂

**資料品質風險**：

## 評審審查 (Adversarial Review)

> **評審立場**：本主題（勞動福祉）最大陷阱 = 乾燥報表 + AI 法律誤導。三提案皆有「以視覺包裝政府公開資料」的傾向，需嚴格區分「真行動工具」vs「Powerpoint 報表」。以下對 7 criteria（受眾真使用、雙北落差、雙北組件 ID 可驗證、AI 邊界明確、Demo 不靠假資料、32hr 可行、原創）逐項勾叉。

### 提案 #1「雙北跨城血汗線」審查

**評分**: 應用 28/40 / 技術 17/30 / 創意 25/30 / **總分 70/100**

**致命問題（≥3）**:
1. **OD 資料根本不存在於 data.taipei**：提案自承「需從通勤普查／健保投保推估」，但實際上雙北 inventory 沒有「行政區×行業」級的 OD dataset。若用主計處「就業者行業結構」+「行政區人口」做 cross-product 推估，會產生**幾何級放大的誤差**（推估誤差可能 30-50%），demo 弧線本質是「設計師畫的故事線」而非真資料 → 違反硬性條件「真實雙北資料」。
2. **「我老闆是不是地雷」AI 功能有嚴重法律責任邊界問題**：違法雇主名單是「歷史公告」，公告後雇主可能已改善；AI 回覆「您的雇主是地雷」可能構成商譽損害 / 不實陳述。即使加浮水印，黑客松 demo 上若被質詢「萬一誤判呢？」直接掉分。
3. **「個別公司不顯示」與「點擊熱區彈出公告 PDF」自相矛盾**：宣稱保護小店家不上地圖，但點熱區能看到公告 = 形同間接公開。可解釋性敘述前後不一。
4. **persona 阿芳「跨城通勤族集中低薪三業」是論述包裝**：提案沒提出實際的因果證據鏈，只用「主計處推估」就把通勤族跟低薪行業綁在一起，可能引發「污名化新北通勤族」的反效果。

**改進建議（≥2）**:
1. **降規模、提誠實度**：放棄「OD 弧線」，改成「雙北通勤者 OD 桑基圖（Sankey）」用主計處實際發布的縣市間就業遷移粗數據；地圖只保留違法雇主熱區（這個是真資料）。
2. **AI 改成「申訴助理」而非「雇主風險判定」**：使用者輸入處境 → AI 只回覆「對應勞基法第 X 條 + 最近申訴窗口」，不對「雇主」做標籤化判斷。降低法律責任，同時保留行動入口。
3. **加入「資料快照日 + 違法已改善通報」雙時間軸**，避免歷史名單被當現役地雷。

**7 criteria 勾叉**:
- 受眾真使用 ✓（通勤族真實存在）
- 雙北落差真實 ✗（OD 推估而非實測）
- 組件 ID 可驗證 ✓（6 個 ID 都在 inventory）
- AI 邊界明確 ✗（地雷雇主標籤有法律風險）
- Demo 不靠假資料 ✗（OD 弧線推估）
- 32hr 可行 △（OD 推估 + 弧線 layer 容易爆時間）
- 原創 ✓（違法雇主空間化確實少見）

**32hr 可行性**: 中偏低。OD 推估管線（主計處 CSV 清洗 → 行政區聚合 → arc-layer 渲染）+ 違法雇主地理編碼（CSV 多無經緯度，需 batch geocoding，data.taipei 違法雇主原資料只有「事業單位地址」字串）+ AI RAG 法條 → 估 40-50hr。地理編碼若超過免費 API 配額會卡住。

**雙北硬性驗證**: 6 組件中 5 組明確雙北雙資料源（OD/違法雇主/年齡/勞檢/資遣解僱/性平），ID 多數可比對 inventory（`A3408B16-...` 新北違法雇主、`13F9C688-...` 新北勞動力、`C285509A-...` 新北就業者年齡、`67e8ddde-...` 北市就服處皆可驗證）。**勞檢密度組件北市 ID 寫成 `24c3dad7-...` 截斷**需補齊。OD 組件雖宣稱用 `18f45255-...` + `13F9C688-...` 但這些是「勞動力統計」非 OD，組件等於是用統計推估「假裝」OD → 雙北條件「形式滿足、實質存疑」。

**Demo-ware 風險**: 高。OD 弧線 = 視覺糖衣，背後是推估；persona 阿芳故事是設計過的。評審若追問「弧線怎麼算的」會破功。

**最終裁決**: **不建議首推**。視覺最炸但真誠度最低；建議降為「政策端附加圖層」融入 #2，捨棄 OD 弧線改用桑基圖。

**評分**: 應用 33/40 / 技術 19/30 / 創意 24/30 / **總分 76/100**

**致命問題（≥3）**:
1. **AI 補助試算錯誤的責任無法承擔**：補助規則每縣市各異、條文常引用上位法（社會救助法、特殊境遇家庭扶助條例），且**收入計算**（家戶月收 / 動產 / 不動產 / 排富條款）規則複雜到連社工都會錯。LLM RAG 把規則 chunk 化後做試算，幻覺風險極高；demo 時若評審輸入刁鑽 persona（「我有房但沒收入」「同住直系血親收入要不要算」）幾乎一定錯。錯了就是市民跑空 → 違反「決策→自動輔助」願景的最後一哩。
2. **新北「政府福利補助資訊」`73C24D1C-...` 是純文字描述，且該 dataset 是「政府對民眾」公告 PDF/文字，並不是結構化補助規則庫**。提案說「離線 LLM 一次性結構化」，但**離線結構化本身就會引入錯誤**，且補助條文每月變更（特別是中央修法後）→ 結構化結果很快過時。
3. **補助分布資料更新慢（多為年資料 / 季資料）**：提案的「補助申請熱區 choropleth」資料來自年度統計，根本沒法反映「現在」狀態；市民端工具用年資料做即時試算 = mismatch。
4. **隱私邊界鬆散**：使用者輸入「單親、低收、月入 22k、家庭組成」這種高敏感屬性，沒有明確說明 (a) 資料儲存策略 (b) cookie/log policy (c) 是否上傳至 LLM provider。Demo 上若有評審問「這份資料會不會被用來訓練？」沒有準備好答案會掉分。

**改進建議（≥2）**:
1. **AI 試算改成「資格篩選器」而非「資格判定」**：AI 只回覆「您『可能』符合 N 項補助 → 請至 X 窗口確認」，**禁止輸出「您不符合 / 您一定符合」這種絕對結論**。所有結果一律標「初篩、非最終認定」。
2. **改用「已知正確規則」白名單**：放棄全 RAG，改成「人工整理 10-15 個最高需求補助的結構化規則 JSON」，AI 只負責 NL → 欄位萃取，規則比對用程式碼判斷（deterministic）。LLM 只在「找不到對應補助」時做 fallback semantic search。
3. **加入「資料快照 + 最後驗證日」並提供「如本資料過期，以官網為準」單鍵跳轉**。
4. **隱私聲明 + 不存儲本機輸入**（前端 only state）。

**7 criteria 勾叉**:
- 受眾真使用 ✓✓（市民端 NL 入口確實高頻需求）
- 雙北落差真實 ✓（補助不同步是真議題）
- 組件 ID 可驗證 △（北市補助 ID 多可驗，新北 `73C24D1C-...`、`F29F882D-...`、`7C280938-...` inventory 內找得到；但「組件 6 知識庫覆蓋率」是 meta 組件，是否計入硬性 4 件需評審認可）
- AI 邊界明確 ✗（試算錯了沒退路）
- Demo 不靠假資料 △（資料真但離線結構化結果可能失真）
- 32hr 可行 △（RAG + 結構化萃取 + 多 persona 試算頗重）
- 原創 ✓（市民 transactional 工具確實稀缺）

**32hr 可行性**: 中。RAG 預建（資料抓取 + chunking + embedding）約 8hr；NL 結構化萃取 prompt + 規則比對 6hr；地圖（社福據點點位 + ISO 同心圓）8hr；前端卡片 UI 6hr；測試 4hr → 32hr 內可行**但前提是 RAG 改用白名單規則**。若硬做全 RAG，超時。

**雙北硬性驗證**: 6 組件雙資料源覆蓋良好（北市 `4a260f77-...` `528b8fdd-...` `dde8fc5f-...` + 新北 `73C24D1C-...` `F29F882D-...` `7C280938-...` `4427DB9F-...` `ABFDE51A-...`），地圖層具備。**唯一風險**：組件 5「住宅補貼」北市側僅寫「相關」沒給 ID → 需補齊。組件 6 是 meta 性質，能否算雙北組件存疑。

**Demo-ware 風險**: 中。NL → 卡片是真功能，地圖也用真據點；但「補助總額對比」若靠 LLM 推算總額，demo 上會看似華麗但任何細節質詢都會破功。

**最終裁決**: **建議首推但需收斂 AI 邊界**。修正 AI 為「初篩 + 跳轉官方」、用白名單規則替代全 RAG，避免法律責任陷阱。

### 提案 #3「銀色 × 障礙就業地圖」審查

**評分**: 應用 30/40 / 技術 21/30 / 創意 22/30 / **總分 73/100**
