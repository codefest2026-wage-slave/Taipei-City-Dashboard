# 食安監控系統 — 題目功能規劃（會議整理）

> 來源：團隊語音會議逐字稿整理
> 性質：題目功能展開 — 從家長視角切入的食安預防系統，2 張地圖 + 3 個輔助組件
> 用途：給隊員 / 評審 / 家長角度的快速理解，落地以 MVP 為先

---

## 專案定位

**食安監控 / 查詢系統**，以「家長角度」為主要敘事入口。核心精神類似治安系統 ——「平常最好不需要用到，但需要的時候就是最關鍵的訊息來源」。提案中還反覆提到要從「預防」切入，而不是「事後告知」。

主要受眾分層：
- **家長**（主）：入學時挑學校、外食時挑餐廳、發生事件時快速掌握
- **學校**（次）：投標前檢視供應商歷史
- **CEO / CTO / 政府單位**（展示用）：透過風險矩陣等視覺化快速掌握全局

整個 Demo 規劃是 **2 張地圖 + 3 個輔助組件** 的結構。

---

## 地圖組件

### 地圖 1：校內食安地圖（校園供應鏈）

這是整場討論最核心的組件，被定義為「主場景」。

**畫面架構**：上方查詢欄 + 中間地圖區 + 下方事件列表 + 側邊 AI 分析面板。

**預設視角**：顯示雙北所有國中小學的位置（以學校為節點），曾發生過食安事件的學校以紅色 / 亮點標示出來。預設不顯示供應鏈連線，避免「廠商連來連去太亂」。

**互動流程設計成兩條路徑**：

第一條路徑（從學校出發，家長視角）：
家長在搜尋欄輸入小孩的國小 → 地圖聚焦該校 → 點擊學校節點 → 拉線顯示這間學校的所有供應商 → 再點某個供應商 → 顯示該供應商「還提供給哪些其他學校」（這就是會議裡反覆強調的 network graph 概念）→ 出問題的供應商以紅色標示。

第二條路徑（從事件出發，新聞視角）：
下方事件列表（由近到遠）→ 點選某個食安事件 → 地圖自動畫出「哪間國小出問題、哪個供應商有問題、這個供應商還提供哪些國小」的影響範圍。

**面板顯示資訊**（點擊節點後）：
- 學校面板：該校歷史食安事件、AI 摘要、原始新聞連結
- 供應商面板：稽查記錄（最近稽查通過時間 / 未稽查記錄 / 未通過記錄）、危害等級（Critical / High / Medium / Low）、改善狀態

**開關控制**：面板上需要有 toggle 控制是否顯示供應鏈連線，以及是否顯示「曾發生事件」的高亮。

### 地圖 2：校外食安地圖（餐廳 / 外食）

故事線銜接：「我們不只在乎校內，你帶小孩外食時也要注意」。

**內容**：全台食品抽驗結果（MVP 範圍鎖定雙北）、發生過食安問題的地點、餐廳的稽查狀態。

**呈現方式**：以區為單位顯示熱點（會議中討論到「以區為單位是重點」），點擊後顯示該餐廳 / 場所的稽查歷史。

**資料優勢**：俊紅在之前的 branch 有做過「差不多完成版本」（已部署於 main 的 dashboard 503），可以直接拉回來改用，不需從零開始。

## 輔助組件

### 組件 3：違規食品類別排行（Rank）

從預防角度切入的組件，**俊紅之前也已有近乎完成的版本**。

**內容**：把過去所有出現問題的食材整理出排行，告訴父母「在外吃時要特別注意哪些類型的食材容易出問題」。

**故事點**：不要等事件發生才告訴你，你應該事前就能知道哪些品項風險高。

### 組件 4：稽查強度折線圖

**內容**：稽查次數 vs 違規率的時序折線圖。

**論點**（會議中很在意這個）：如果政府一年只抽查一件就宣稱「通過率 100%」是沒意義的，所以要呈現「效度」—— 也就是抽了多少、抓到多少。X 軸是時間，Y 軸是檢驗數量與違規率。

### 組件 5：風險矩陣（Risk Matrix）

**這個是討論到一半臨時加上去的**，被笑稱「越來越 AI 的感覺」，但團隊一致認為政府單位、CEO / CTO 喜歡這種快速掌握的視覺化。

**設計**：四象限矩陣
- X 軸：一年內是否被檢測出違規
- Y 軸：一年前是否被檢測出違規

**四個象限的解讀**：

| | 一年前 有違規 | 一年前 無違規 |
|---|---|---|
| **一年內 有違規** | 高危險店家 | 新興風險（最近才開始出問題）|
| **一年內 無違規** | 改善中（可能已改善）| 優良店家 |

## 底層資料處理（支援上述組件）

雖然不是直接的「展示組件」，但會議花了大量時間討論這幾個資料管線。

### 新聞處理管線

**資料來源優先序**：
1. 政府公文 / 衛生署食安公告（優先，可信度高、不會被帶風向）
2. 一般新聞媒體（備用，作為輔助驗證 ——「議事 = 新聞給的、確認事實 = 政府給的」）

**LLM 處理**：
- 抽 Tag：食物類型 Tag、學校 / 地點 Tag、確切性等級
- 二次過濾：處理「新聞提到台北但事件其實發生在台南」這類關鍵字檢索誤判
- 生成 AI Summary（在入庫時就先生成，前端讀取時直接顯示，不再即時呼叫 LLM）

**DB 結構**：`title` / `content` / `summary` / `tags` / `status`

**事件詳情欄位**（用於事件面板）：發生時間、死亡人數、受傷人數、住院中人數、確認問題食物、議事問題食物。

### 供應商與違規資料

- **校內供應鏈來源**：每間國中小的團膳供應商清單（只篩有團膳的學校，不要全撈避免 LLM 跑爆）
- **供應商稽查資料**：台北市政府有，新北市較難取得（會議後段發現新北的稽查資料是用 dotnet ASPX，逆向工程後發現底層其實有完整的 JSON link 可拉）
- **違規清單分類**：拆成「個人 / 農場」與「商業業者」兩份 CSV
- **危害等級**：用 LLM 從法條 + 產品角度評估，標準化分為 Critical / High / Medium / Low
- **MVP 假資料策略**：因 2024–2026 年國中小食安事件樣本少，會自製 3–5 份模擬新聞稿（HTML）讓系統去攝取，demo 時說明「真實資料存在於某些區域，我們先串假資料來呈現流程」

## 完整 Demo 故事線

把所有組件串起來會是這樣的敘事：

**情境**：我是家長，小孩要入學。

1. 開啟系統 → **校內地圖**搜尋小孩的國小 → 看到這間學校的供應商歷史是否乾淨
2. 點擊學校 → 拉出供應鏈 → 發現某供應商出過事 → 紅色亮起 → 該供應商還提供給其他幾間學校（network 概念）
3. 切換到 **校外地圖** → 帶小孩外食時，查詢附近餐廳是否曾抽驗不合格
4. 看 **違規食材 Rank** → 知道哪些品項風險高，事前避開
5. 看 **稽查強度折線圖** → 證明資料的「效度」（抽得夠多才有意義）
6. 看 **風險矩陣** → 一眼判斷店家是高危、新興、改善中、還是優良

最後的口號是：**「零容忍」**。

## ⚠️ MVP 收斂建議

逐字稿中段有一句評語很關鍵：

> 「你們現在有點像是想在做之前就先把完整的路徑討論出來，但以黑客松還有 MVP 來講，這樣其實不適合。」

也就是說，這份規劃雖然完整，但落地時建議用：

**校內地圖（主）+ 違規 Rank（已有舊版本）+ AI Summary** 作為最小可行展示

風險矩陣和稽查折線圖視時間餘裕再加。**隔天 9:30（最晚 10:00）就要交檔案**。
