## 二、評分方法說明

### Phase 3 五維評分（1–10分）
| 維度 | 說明 | 權重 |
|------|------|------|
| 政府需求 | 公務員會週/月實際使用嗎？ | 25% |
| 市民有感 | 市民是否能直接感受改善？ | 15% |
| 雙北必要 | 「必須是雙北」的論證強度 | 25% |
| 可行性 | 24h完成 + 資料兩邊都有 + 合規 | 20% |
| 創意差異 | 與官方儀表板已實作項目的距離 | 15% |

### Phase 4 競爭重疊度（0–10）
0 = 沒人想到這個切角；10 = 撞題，會有很多隊做。≥ 6 直接淘汰。

### Phase 5 資料可行性
✅ = 雙北資料均已確認可取得且可分流
⚠️ = 有缺口但有替代方案
❌ = 核心資料不可分流，不可行

## L-01-1｜複查優先佇列引擎

> **主題**：勞動福祉｜**評分**：8.2／10｜**競爭重疊度**：3／10｜**資料可行性**：✅ 已驗證｜**賽前補強難度**：中

### 🎯 一句話定位
「身為台北市勞動局勞檢員，我現在可以在每季複查出發前，直接看到風險排序後的前 20 家雇主清單，以及 AI 說明每家為什麼排這裡，不再靠經驗猜或翻舊紀錄。」

### 🔍 解決什麼問題
台北市有 15,007 筆、新北市有 14,155 筆的違法雇主歷史紀錄，每季複查時勞檢員人力有限，只能挑部分雇主複查，但目前無任何系統幫助決定「先去哪家」。現行做法依賴個人經驗或隨機抽選，導致高風險慣犯雇主可能被跳過。

### 👥 核心使用者
- **主要**：台北市/新北市勞動局勞工檢查員 — 每季複查排程時使用，每週開會時用來討論優先順序
- **次要**：勞動局主管 — 季報簡報時用來說明複查重點

### 📊 決策輸出
**本季複查優先 TOP 20 雇主清單**，每筆包含：
- 雇主名稱、統編、地址、行業別
- 綜合風險評分（滿分100）
- AI 解釋：「為什麼這家排第一」（例：連續 3 年違規、員工規模 50 人以上製造業、上次罰款後 8 個月未複查）
- 權重滑桿：可調整「違規頻率/員工規模/違規類型嚴重性」比重

### 🏙️ 雙北差異化邏輯
| | 臺北市 | 新北市 |
|---|---|---|
| 主要違規類型 | 工時超時、未給加班費（服務業） | 職安未落實、未足額投保（製造業） |
| 資料結構 | 中文欄位、民國日期格式 | 英文欄位、西元日期、含統編（可直接 JOIN GCIS） |
| 切換後改變什麼 | 排序權重調整（工時違規加重）；AI 解釋語境切換為「製造業職安」 |

> **切換邏輯**：下拉切換後，風險評分公式中的違規類型權重自動調整，AI 提示語境也跟著換，確保台北/新北各自最相關的違規面向被優先考量。

### 🤖 AI 模型角色（llama3.3-FFM-70B）
- **SQL+圖表做不到的事**：SQL 可算出「違規次數排名」，但無法解釋「為什麼這間比那間風險更高」——尤其當兩間違規次數相同但行業/規模/類型不同時，排序理由需要跨欄位語意推理。
- **AI 真正做的事**：接收單一雇主的結構化風險特徵（違規次數、距上次複查天數、行業別、員工規模、違規類型），生成自然語言解釋段落。
- **Prompt 方向**：「請根據以下雇主違規紀錄，以勞檢員視角說明為什麼這家排在複查優先清單第一名，並指出最需要重點稽查的項目：[JSON資料]」

### ❓ 自我攻擊（評審最可能問的三個問題）

**1. Q：這跟現有勞動部查詢系統有什麼不同？**
A：勞動部系統是「查詢工具」——你知道一家公司名稱，才能查它有沒有違規。我們做的是「排序引擎」——主動告訴你下週應該去哪 20 家，並說明為什麼。輸出物是可執行的行動清單，不是被動查詢介面。

**2. Q：24 小時內做得完嗎？**
A：可以。核心資料（雙北 29,162 筆）已在本地，ETL 約 3h，評分引擎約 2h，前端約 4h，AI 串接約 1h，地圖組件約 2h。共 12h，在 24h 內可完成完整版 Demo，含雙北切換邏輯。

**3. Q：加權公式是誰決定的？有學術依據嗎？**
A：這是本構想的專屬死穴。目前公式（違規次數×0.4 等）是參考勞動部「勞動條件一般檢查實施計畫」的稽查重點方向設計，但無嚴格學術引用。應對：前端提供「權重滑桿」讓使用者自行調整，定位為「決策輔助工具」而非「絕對排名」，主動說明「建議勞動局主管依實際業務需求調整權重」。

## L-04-1｜就業資源可及性缺口地圖

> **主題**：勞動福祉｜**評分**：8.0／10｜**競爭重疊度**：5／10｜**資料可行性**：✅ 已驗證｜**賽前補強難度**：低

### 🎯 一句話定位
「身為新北市就業服務員，我現在可以一眼看出哪些里的弱勢勞工距最近就服站超過 3 公里、服務真空最嚴重，不再靠直覺猜測哪裡最需要加派人力。」

### 🔍 解決什麼問題
就業服務站分布依歷史設置，從未系統性對應弱勢人口分布。部分弱勢勞工集中的里距最近就服站超過 3km，步行不可及、無大眾運輸連結，形成服務真空。目前無任何工具量化此缺口，資源補充決策依賴個別承辦的主觀印象。

### 👥 核心使用者
- **主要**：台北市/新北市就業服務中心主任、就服員 — 每月討論轄區服務缺口時使用
- **次要**：社工、里幹事 — 外展訪視路線規劃參考

### 📊 決策輸出
**里級就業資源可及性缺口熱力圖**，輸出：
- 深紅里：弱勢人口密度高 × 就服站可及性低（建議優先設立服務據點）
- 每里點擊後顯示：距最近就服站距離、轄區低收入/中低收入人口數、AI 推薦是否新設及選址理由
- 「新設點推薦」功能：輸入預算可設 N 個新點，AI 推薦最佳座標

### 🏙️ 雙北差異化邏輯
| | 臺北市 | 新北市 |
|---|---|---|
| 弱勢人口分布 | 大安/中正資源豐富，萬華/大同/中山相對弱勢 | 郊區里（三峽、石碇、坪林、烏來）服務真空最嚴重 |
| 就服站密度 | 12 站，分布相對均勻 | 10 站，但轄區面積大，郊區覆蓋率低 |
| 切換後改變什麼 | 缺口模式不同：台北是市中心內部失衡；新北是都市/郊區落差 |

> **切換邏輯**：同一套可及性算法，但台北版以 1km 為服務半徑（步行可及），新北版自動升為 3km（依賴交通工具），反映城市型態差異。

### 🤖 AI 模型角色（llama3.3-FFM-70B）
- **SQL+圖表做不到的事**：SQL 可算出「距離超過 X 公里的里」，但無法根據多因子（距離+弱勢人口規模+交通可及性+現有站負荷量）生成「建議在這裡設站的理由」。
- **AI 真正做的事**：接收里的多因子數據，生成「為什麼這個里需要優先新設就服站」的自然語言推薦理由，可直接貼入預算申請書。
- **Prompt 方向**：「請根據以下里的就業資源可及性數據，以就業服務中心主任視角，說明為何此里應優先新設服務據點，並建議選址邏輯：[JSON資料]」

## ENV-E5-1｜稽查強度 × 空品相關性儀表板

> **主題**：永續環境｜**評分**：7.1／10｜**競爭重疊度**：5／10｜**資料可行性**：✅ 已驗證｜**賽前補強難度**：低

### 🎯 一句話定位
「身為台北市環保局空污防制政策評估承辦，我現在可以用折線圖直接看出稽查密集月份與空品改善的相關性，讓議會質詢時的政策效益說明有了數據支撐。」

### 🔍 解決什麼問題
環保局每年投入大量資源在固定污染源稽查，但稽查行動是否真的改善空氣品質，從未被系統性量化。現行績效報告只說「稽查了 X 件」，無法回答「稽查有沒有用」這個核心政策問題。

### 👥 核心使用者
- **主要**：台北市/新北市環保局政策評估承辦、空污防制科 — 年度績效報告撰寫、議會質詢備稿時使用
- **次要**：議員助理 — 查詢稽查效益時使用

### 📊 決策輸出
**稽查強度 × 空品相關性分析儀表板**，輸出：
- 月別稽查件數折線 + PM2.5 月平均折線並排
- 相關係數（Pearson r）及顯著性說明
- AI 政策摘要：「2024Q2 稽查密集期間 PM2.5 下降 12%，相關係數 0.67，但氣象因子（降雨偏多）可能共同影響，建議以控制氣象條件的偏相關作更精確分析」

### 🏙️ 雙北差異化邏輯
| | 臺北市 | 新北市 |
|---|---|---|
| 稽查資料 | 台北月別稽查件數完整，空品測站數多 | 新北工業區稽查資料，但空污來源更複雜（跨境）|
| 分析意義 | 服務業/交通污染為主，稽查效果相對直接 | 工業固定源為主，稽查效果延遲性高 |
| 切換後改變什麼 | 相關係數解讀框架不同；新北版需說明「工業污染控制效果需較長時間顯現」|

### 🤖 AI 模型角色（llama3.3-FFM-70B）
- **SQL+圖表做不到的事**：相關係數計算統計套件可做，但「為什麼相關性在某個季節特別高/低、這個數字應如何向議員解釋、有哪些混淆因子需要說明」的自然語言政策摘要無法靠 SQL 生成。
- **AI 真正做的事**：接收相關係數分析結果，生成「政策語言版」摘要，包含主動說明「相關不等於因果」的學術誠實說明（反而是加分點）。
